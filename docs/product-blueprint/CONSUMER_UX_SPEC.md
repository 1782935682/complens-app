# 消费者体验规范（CONSUMER_UX_SPEC）

> 本文属于 `docs/product-blueprint/` 蓝图集，补充 `CONSUMER_DECISION_SPEC.md` 的界面表达规则。目标是让 CompLens / 成分镜 更像普通消费者的食品标签解读工具，而不是专业法规查询系统。

## 1. 体验原则

1. **先回答“我该看什么”**：默认展示关注项、重点提醒、配料/营养重点，不把法规表格放在第一屏。
2. **专业信息可展开**：数据来源、可信等级、GB2760 证据、JECFA 说明保留，但默认折叠到“数据来源和查看依据”。
3. **用户拍的不一定是配料表**：入口和流程必须支持配料表、营养成分表、包装正面和未知标签。
4. **关注项驱动排序**：报告页优先展示与用户关注项相关的内容，而不是固定展示所有专业字段。
5. **只做标签信息参考**：不做购买命令、医疗建议、营养诊断或绝对健康判断。

## 2. 首页体验

首页主入口文案：

```text
拍照解读食品标签
```

首页场景卡片：

- 控糖看配料
- 低钠看营养
- 给孩子买零食
- 过敏 / 忌口检查
- 少添加选择
- 对比两款食品

首页不应把“添加剂查询”作为第一主入口。搜索仍可保留，但只能作为辅助入口。

## 3. 拍照体验

前端应提示三类拍摄目标：

- 拍配料表：看成分和添加剂。
- 拍营养成分表：看糖、钠、脂肪、热量。
- 拍包装正面：核对 0 糖、低脂、高蛋白等卖点。

MVP 不要求三张都拍：

- 一张也能分析。
- 两张可以合并。
- 三张报告更完整。

系统无法判断标签类型时，必须让用户选择：

- 这是配料表。
- 这是营养成分表。
- 这是包装正面。
- 都不是，重新拍。

### 3.1 后续自动识别分流体验（P1/P2）

后续支持条码和数字标签二维码后，首页 / 扫描页仍保持现有一个拍照入口，原有 UI 不需要改成三个入口。用户拍食品标签 / 包装后，由后端自动识别：

- 商品条码。
- 数字标签二维码。
- 配料表 / 营养成分表。

交互约束：

- 不新增 `扫商品条码`、`扫数字标签二维码`、`拍配料表 OCR` 三个外露主入口。
- 商品条码分支只负责按 GTIN 查询商品及配料数据。
- 数字标签二维码分支只负责解析数字标签页面中的配料和营养信息。
- 配料表 OCR 分支只负责识别包装配料表文字，并继续进入文本确认。
- 任一分支信息不足时，只提示“可以重新拍摄包装其他区域或手动补充”，不自动跳转、不自动发起其他分支、不把三种识别方式设计成串行流程。
- 各分支拿到的数据都进入同一套成分归一化、官方成分知识库匹配和成分分析流程。

## 4. 文本确认体验

OCR 文本确认页不只确认配料表，也要支持营养成分表和包装正面文案。页面必须显示：

- 当前识别类型。
- 可编辑 OCR 文本。
- 修改标签类型入口。
- 继续分析按钮。
- 重新拍 / 补拍入口。

OCR 输出仍是用户输入来源，不是权威数据。页面不得出现购买建议或健康判断。

## 5. 我的关注项体验

“我的关注项”是主路径的一部分。首次进入可引导用户选择，也可允许跳过。

MVP 本地保存，不要求登录。关注项包括：

- 控糖
- 低钠
- 减脂
- 高蛋白
- 少添加
- 给孩子看
- 过敏 / 忌口

后续关注项：

- 素食
- 清真
- 宗教忌口

关注项设置页应支持：

- 多选关注目标。
- 添加忌口成分。
- 删除或关闭关注项。
- 本机保存和清空。

## 6. 报告页体验

报告页名称：**食品标签解读**。

报告默认结构：

1. 一句话结论。
2. 我的关注项。
3. 重点提醒。
4. 配料表解读。
5. 营养成分解读。
6. 包装卖点提示。
7. 食品添加剂分组。
8. 过敏 / 忌口提示。
9. 暂未识别 / 暂未收录。
10. 数据来源和查看依据。

信息层级：

- 第一屏：普通消费者语言、关注项命中、建议查看点。
- 第二层：配料、营养、卖点提示分组。
- 第三层：数据来源、可信等级、原文证据、法规/评价来源。

信息不足降级：

- 只识别到生产日期、喷码、正面卖点或少量营养数字时，不展示完整消费建议。
- 营养表有效数字少于 2 项且没有配料表时，默认进入补拍 / 手动补充。
- 不把孤立数字强行归到钠、糖、脂肪、蛋白质等字段。
- 第一屏直接提示“信息不足”，并给出补拍配料表、补拍营养表或手动补充的动作。
- 识别到的少量线索可以短句展示，但其余未识别营养项不铺满页面。

## 7. 话术规范

推荐：

- 建议关注糖类配料和钠含量。
- 部分人群可能需要留意甜味剂。
- 信息不足，请补充标签文字后查看结果。
- 仅供标签信息参考，不构成医疗或营养诊断。

禁止：

- 可以买
- 不能买
- 健康
- 不健康
- 安全
- 有害
- 致癌
- 治疗
- 诊断
- 一定过敏

## 8. 专业信息展示

专业信息默认隐藏或折叠：

- GB2760 使用范围和限量。
- JECFA 安全评价。
- staging / pending_review 状态。
- sourceHash / sourcePage / rawSourceText。
- 导入批次和审核记录。

但专业信息必须可追溯：

- 在“数据来源和查看依据”中展示。
- 对 `pending_review` 明确标注“待复核”。
- 对 `unknown_from_ocr` 明确标注“暂未收录”。
- 对 AI 解释标注“AI 生成，仅做解释层”。

## 9. 两款商品对比体验（后续）

产品对比页只按标签信息和用户关注项做比较：

- 糖更低。
- 钠更低。
- 脂肪更低。
- 蛋白质更高。
- 食品添加剂数量更少。
- 用户关注项命中更少。

禁止输出“更健康 / 不健康”。输出应使用“更符合你当前关注项”。
