# 市面优秀食品扫描 / 营养 App 调研与成分镜优化清单（2026-06-19）

本文件把市面上成熟的食品扫描、营养记录和成分解读类 App 做成产品改进依据。结论服务于成分镜当前方向：**微信小程序单端优先，拍食品标签 → OCR → 文本确认 → 大白话解读报告 → 历史复用**。

调研口径：

- 优先参考官方页面、官方功能说明和公开可查资料；无法核实的中文小程序传闻不作为结论依据。
- 关注功能结构、用户路径、结果表达、留存机制和数据可信呈现，不评价其医学或营养正确性。
- 成分镜不照搬“买 / 不买”式判断，不做医疗化结论，不把不确定数据包装成确定结果。

## 1. 竞品拆解

| 产品 | 核心功能 | 做得好的地方 | 成分镜可学习 | 不照搬 |
|---|---|---|---|---|
| Yuka | 扫码识别食品 / 化妆品，给出颜色化评分、扣分原因、历史和相似替代项 | 一次扫描立刻得到结果；颜色和分数极易理解；会解释“为什么”；历史和个性提醒增强复用 | 第一屏要直接给“结果摘要 + 前 3 个重点”；添加剂、糖、钠等原因要一句话讲明；历史和分享要轻 | 不做绝对评分；不输出购买导向替代；不把复杂饮食问题压成单一好坏 |
| Open Food Facts | 开放商品库，扫码查看配料、营养、过敏原、添加剂、Nutri-Score、NOVA、Eco-Score，支持用户补充照片和数据 | 商品库和开放数据生态强；条码是低摩擦入口；数据可复用，可做对比和纠错 | P1 再接条码 / 数字标签分支；保留来源、可信状态和用户纠错入口；商品库数据不得抢 P0 OCR 主路径 | 不为了商品库完整度拖慢小程序第一版；众包数据不能当成已确认结论 |
| CodeCheck | 扫码或搜索查看食品 / 日化成分、营养、认证标签、糖脂信息、过敏 / 素食 / 棕榈油等关注项 | 把“我在意什么”变成命中提示；扫描、搜索、OCR 多入口互补；不同关注项有清晰标识 | “我的关注项”要成为报告排序规则，而不只是设置页；过敏、控糖、儿童等命中要高亮但保持中性 | 不扩到化妆品 / 日化；不采用泛化风险算法替代食品标签解读 |
| Fooducate | 扫码、食物等级、宏量营养、饮食偏好、记录、社区和食谱 | 不是只记录数字，会解释食物等级和隐藏成分；用户能按低碳、素食等偏好个性化 | 添加剂大白话要解释“它在标签里通常扮演什么角色”；搜索结果要回答普通人的实际问题 | 不把成分镜变成热量日记或社区产品 |
| MyFitnessPal | 食物记录、卡路里 / 宏量营养、条码、语音 / 拍照记录、目标、进度和多设备同步 | onboarding 先问目标；大数据库 + 快速记录降低长期使用成本；进度和历史强化留存 | 首次使用先选控糖 / 儿童 / 过敏；历史页要支持快速重看、收藏和按关注项筛选 | 不把每日打卡、减重目标和运动同步做成主路径 |
| Cronometer | 条码 / 食物记录、最多 84 项营养素、图表、报告、专业版和高可信数据 | 数据细，但默认能用图表消化；“可信数据”是核心卖点；专业信息分层 | 营养字段用条形图 / 状态短句展示；来源和可信提示放在可展开层 | 不把普通用户报告做成营养师仪表盘 |
| Lose It! / Foodvisor 类拍照记录 | 拍食物估算、记录餐食、卡路里和目标 | 拍照比手输轻；识别后仍允许用户修正 | 成分镜拍照后必须有清晰的识别阶段、缺失提示和快速修正 | 食物识别不能替代食品标签 OCR；估算不能当作标签事实 |

主要来源：

- Yuka：公开报道显示其扫码、评分、扣分原因、替代项、历史、订阅和品牌独立性是核心体验；同时也存在“过度简化”的争议。参考 [Wired](https://www.wired.com/story/yuka-app)、[Washington Post](https://www.washingtonpost.com/climate-environment/2026/03/17/yuka-app-processed-food-health/)、[Yuka 公开资料](https://fr.wikipedia.org/wiki/Yuka_%28application%29)。
- Open Food Facts：开放商品库、条码、配料、过敏原、添加剂、营养、Nutri-Score / Eco-Score、用户贡献和商品对比。参考 [Open Food Facts 公开资料](https://en.wikipedia.org/wiki/Open_Food_Facts)。
- CodeCheck：扫码 / 搜索、成分和营养值、认证标签、糖脂、过敏 / 乳糖 / 麸质 / 棕榈油等关注项、机器文字识别。参考 [CodeCheck 公开资料](https://en.wikipedia.org/wiki/CodeCheck)。
- Fooducate：食物等级、扫描、宏量营养、饮食偏好、记录和社区。参考 [Fooducate 官方页](https://www.fooducate.com/)。
- MyFitnessPal：条码、语音 / 拍照记录、目标、进度、超过 1800 万 / 2000 万级食品库、多设备同步。参考 [MyFitnessPal 官方页](https://www.myfitnesspal.com/)。
- Cronometer：条码、营养追踪、最多 84 项营养素、报告图表、数据可信。参考 [Cronometer 官方功能页](https://cronometer.com/features/)。

## 2. 市场共性结论

1. **入口越单一，越容易启动。** 成熟产品都把“扫 / 拍 / 记”放在第一动作，其他功能围绕结果服务。成分镜首页继续坚持一个拍食品标签主入口，搜索、设置、历史都不能抢焦点。
2. **结果要先给答案，再给理由。** 普通用户不会先读数据库字段。报告第一屏必须是结果摘要、重点提醒、营养数字和添加剂解释，详细来源放到二层。
3. **颜色和分数能降低理解成本，但容易制造误解。** 成分镜可以用轻量状态和条形图帮助扫读，但不做“一分定性”的总评分。
4. **个性化比专业数据更有价值。** 控糖、儿童、过敏这类关注项必须影响排序、提醒和历史筛选，而不是只作为用户资料存在。
5. **解释要回答“为什么看”。** 用户看到甜味剂、防腐剂、增稠剂、含钠配料时，最需要的是它在食品里通常做什么、为什么会被提醒、和自己的关注项有什么关系。
6. **历史是留存核心。** 成分镜不需要先做复杂社区，但必须让用户能快速回看、收藏、分享摘要，并在超市里复用上次经验。
7. **数据可信要分层，不要把责任甩给用户。** 可以展示来源和可信状态，但报告本身必须给出当前可用结果；信息不足时明确缺哪些，不硬凑完整结构。
8. **条码 / 商品库是加速器，不是第一版地基。** 商品库能显著提高速度，但国内商品覆盖、数字标签、来源授权和更新都需要治理，适合 P1 作为拍照入口下的自动分支。

## 3. 成分镜优化清单

### P0：现在就应继续收口

1. 首页只保留一个强主动作：拍食品标签；辅以最近一次结果、手动输入和历史入口。
2. 拍照页增加“目标区域质量提示”：模糊、反光、太远、文字倾斜、目标字段不足时，给出下一步动作。
3. OCR 后确认页改成结构化确认：配料、营养数字、致敏原、包装声明分区展示；用户只修正识别缺失或明显错误的部分。
4. 报告第一屏固定为：结果摘要、前 3 个重点、营养数字条形图、添加剂大白话、未确认线索数量。
5. 添加剂解释统一成三段短信息：标签写法、通常作用、为什么在本次被提醒。
6. 搜索结果保持辅助定位：名称、类型、为什么看、标签写法、数据来自、结果用途；不展示专业字段堆栈。
7. 历史页强化复用：收藏、最近查看、按关注项筛选、从历史生成分享摘要。
8. 产品输出审查继续拦截空话、购买导向、医疗化表达、已废弃口径、非目标字段进入报告。

### P1：小程序验证后增加

1. 在同一个拍照入口下做自动分流：条码、数字标签二维码、配料表 OCR、营养表 OCR。
2. 条码 / 数字标签数据进入报告前必须带来源和可信状态；众包或未确认数据只作为线索。
3. 比一比：同类两款食品按用户关注项并排展示差异，不输出购买结论。
4. 家庭档案：儿童、控糖、过敏 / 忌口可分别保存，报告按当前档案排序。
5. 纠错反馈：用户能标记“识别错了 / 漏了 / 解释不清楚”，形成样本回归队列。
6. 分享卡片：只显示结果摘要、前 3 个重点和未确认数量，不展示长文报告。

### P2：成熟后再做

1. AI 解释生僻成分，只做通俗解释，不作为数据来源。
2. 历史趋势：最近常见的添加剂类型、糖 / 钠 / 脂肪线索变化。
3. 离线 / 弱网队列：先保存图片和识别文字，网络恢复后补全报告。
4. 更多人群模板：孕期、健身增肌、低脂等，必须重新审查文案边界。

## 4. 明确不做或暂不做

- 不照搬 Yuka 式总评分、红绿好坏和替代购买建议。
- 不把成分镜做成热量日记、运动记录或社区产品。
- 不默认展示专业法规表、限量表、数据库字段和复杂来源链。
- 不把化妆品、护肤品、药品或日化规则混入食品标签。
- 不把 AI、OCR、待复核、众包或疑似匹配内容展示成已确认结论。
- 不在信息不足时生成完整但似是而非的报告结构。

---

## 5. 本轮产品决策

1. 成分镜第一版继续坚持“拍食品标签 → OCR → 文本确认 → 解读报告 → 历史”，不因竞品扫码能力强而提前改成商品库主导。
2. 报告优化的第一目标不是更专业，而是让普通人第一眼知道本次最该看哪几项，以及为什么。
3. 搜索是辅助，不是主路径；搜索结果要像报告一样回答用户问题，不展示数据库后台视角。
4. 市场上有效的能力按顺序吸收：个性化关注、历史复用、条形图、纠错反馈、条码 / 数字标签自动分支、比一比。
5. 所有外部产品的评分、替代推荐和“更好选择”机制，都只能作为交互参考，不进入成分镜的结论口径。
