# 消费者食品标签解读与消费决策规范

> 本文属于 `docs/product-blueprint/` 蓝图集，是 CompLens / 成分镜 面向普通消费者的产品决策规范。它定义“用户为什么拍、拍什么、看什么、如何输出”，并约束产品不得退化成专业法规查询或添加剂数据库工具。

## 1. 产品定位

CompLens / 成分镜 是**面向普通消费者的食品标签拍照解读与消费决策助手**。

核心价值：

```text
拍一下食品包装，帮普通消费者看懂配料、营养、卖点和需要关注的地方。
```

用户核心问题：

- 这个食品适不适合我当前的关注目标？
- 我买之前需要看哪些点？
- 这款和另一款比，哪个更符合我的关注目标？

产品可以帮助用户做“购买前 / 食用前的信息整理和关注提示”，但不能替用户做绝对购买判断、医疗判断或营养诊断。

## 2. 文案边界

禁止文案：

- 可以买
- 不能买
- 健康
- 不健康
- 安全
- 有害
- 致癌
- 治疗
- 诊断
- 一定过敏

推荐文案：

- 建议关注
- 更适合重点查看
- 部分人群可能需要留意
- 信息不足，建议结合包装原文确认
- 仅供标签信息参考，不构成医疗或营养诊断

## 3. 消费者核心场景

### 场景 1：超市 / 便利店购买前快速判断

用户状态：手里拿着食品，想快速知道有没有需要关注的成分。

用户动作：

- 拍包装
- 拍配料表
- 拍营养成分表

产品输出示例：

- 这款食品需要重点看：糖、钠、甜味剂。
- 识别到 18 项配料，其中 5 项食品添加剂。
- 营养表显示钠含量需要留意，具体以包装标示为准。

### 场景 2：控糖人群看食品

用户问题：

- 有没有糖、代糖、糖浆？
- 营养表里碳水 / 糖高不高？
- 包装写 0 蔗糖是否还含其他甜味来源？

产品应识别：

- 白砂糖
- 果葡糖浆
- 麦芽糖浆
- 葡萄糖浆
- 蜂蜜
- 浓缩果汁
- 甜味剂
- 碳水化合物
- 糖

输出方式：

```text
控糖人群建议重点查看：糖类配料、甜味剂、碳水化合物。
```

禁止输出：

```text
糖尿病人可以吃 / 不能吃
```

### 场景 3：低钠人群看食品

用户问题：

- 钠高不高？
- 有没有盐、酱油、味精、含钠添加剂？

产品应识别：

- 钠
- 食用盐
- 谷氨酸钠
- 碳酸氢钠
- 磷酸盐
- 酱油粉
- 调味料

输出方式：

```text
低钠人群建议重点查看钠含量和含钠配料。
```

### 场景 4：给孩子买零食

用户问题：

- 有没有甜味剂、色素、防腐剂、咖啡因？
- 配料是不是太复杂？

输出方式：

```text
家长可能会关注：甜味剂、色素、防腐剂、咖啡因。
```

禁止输出：

- 儿童不能吃
- 不适合儿童

除非有明确法规和产品类型依据，否则只能做关注提示。

### 场景 5：过敏 / 忌口人群

用户问题：

- 有没有乳、大豆、坚果、麸质、蛋、海鲜？
- 有没有我自己设置不想吃的成分？

输出方式：

```text
识别到疑似忌口项：大豆。请以包装过敏原提示和配料表为准。
```

禁止输出：

- 无过敏风险
- 一定会过敏

### 场景 6：少添加 / 配料简单人群

用户问题：

- 这个配料表复杂吗？
- 添加剂多不多？
- 有没有食用香精、色素、防腐剂？

产品输出：

```text
配料数量：18 项
食品添加剂：5 项
配料复杂度：较多，建议查看添加剂分组
```

复杂度只作为信息提示，不作为健康判断。

### 场景 7：两款商品对比

用户问题：

- 这两款哪个糖更低？
- 哪个钠更低？
- 哪个添加剂更少？
- 哪个更适合我的关注目标？

后续功能规划：

```text
拍 A 产品标签
拍 B 产品标签
对比配料数量
对比添加剂数量
对比糖/钠/脂肪/蛋白质
对比用户关注项命中数量
```

输出方式：

- A 的钠更低。
- B 的食品添加剂数量更少。
- A 更符合你当前的低钠关注项。

禁止输出：

- A 更健康
- B 不健康

## 4. 识别对象

拍照对象从单一配料表扩展为：

1. 配料表。
2. 营养成分表。
3. 包装正面卖点。
4. 条形码 / 产品名，后续。

OCR 类型：

| labelType | 含义 | MVP |
|---|---|---|
| `ingredient_list` | 配料表 | 必须 |
| `nutrition_facts` | 营养成分表 | 必须 |
| `front_claims` | 包装正面卖点 | 后续 |
| `barcode_or_product` | 条码 / 产品名 | 后续 |
| `unknown_label` | 未知标签 | 必须有降级 |

如果系统无法判断，前端必须允许用户选择：

- 这是配料表。
- 这是营养成分表。
- 这是包装正面。
- 都不是，重新拍。

## 5. 三图组合解读流程

更贴近真实消费场景的流程：

```text
拍包装正面，可选
拍配料表，推荐
拍营养成分表，推荐
→ OCR
→ 用户确认
→ 自动识别标签类型
→ 合并生成食品标签解读
```

MVP 不要求必须拍三张：

- 拍一张也能分析。
- 拍两张可以合并。
- 拍三张报告更完整。

前端提示：

- 拍配料表：看成分和添加剂。
- 拍营养成分表：看糖、钠、脂肪、热量。
- 拍包装正面：核对 0 糖、低脂、高蛋白等卖点。

## 6. 食品标签解读报告结构

报告页名称：**食品标签解读**。

不要叫：

- 法规分析报告
- 风险分析报告
- 添加剂合规报告

报告结构：

1. 一句话结论。
2. 我的关注项。
3. 购买前建议关注。
4. 配料表解读。
5. 营养成分解读。
6. 包装卖点核对。
7. 食品添加剂分组。
8. 过敏 / 忌口提示。
9. 暂未识别 / 暂未收录。
10. 数据来源和查看依据。

一句话示例：

```text
这款食品识别到 18 项配料，含 5 项食品添加剂；如果你关注控糖和低钠，建议重点查看糖类配料和钠含量。
```

不要给绝对购买建议。专业信息默认隐藏，普通人报告默认展示；用户需要时再展开查看法规来源、数据库依据和原文证据。

## 7. 我的关注项

“我的关注项”是核心功能，不是后期边缘功能。MVP 可以本地保存，不需要登录。

关注项包括：

- 控糖
- 低钠
- 减脂
- 高蛋白
- 少添加
- 给孩子看
- 过敏 / 忌口
- 素食 / 清真 / 宗教忌口，后续

细分关注成分：

- 糖
- 糖浆
- 甜味剂
- 钠
- 食用盐
- 防腐剂
- 色素
- 食用香精
- 反式脂肪
- 乳制品
- 大豆
- 坚果
- 麸质
- 蛋类
- 海鲜
- 咖啡因

报告页要根据用户关注项排序，而不是固定展示所有信息。

## 8. 包装卖点核对（后续）

识别包装正面文案：

- 0 蔗糖
- 0 糖
- 低脂
- 高蛋白
- 无添加
- 儿童
- 粗粮
- 代餐
- 低卡
- 非油炸

结合配料表和营养成分表提示：

- 标注 0 蔗糖，但建议继续查看是否含其他糖类或甜味剂。
- 标注高蛋白，请结合营养成分表中的蛋白质含量查看。
- 标注低脂，请结合每 100g 脂肪含量查看。
- 标注无添加，请结合配料表中的食品添加剂查看。

只做“核对提醒”，不做打假结论。

## 10. MVP 与后续边界

MVP 从“配料表识别 + 添加剂查询”调整为：

```text
拍食品标签 → 识别配料表/营养成分表 → 我的关注项 → 普通人报告 → 保存历史
```

MVP 必须包含：

- 配料表识别。
- 营养成分表识别。
- 文本确认。
- 配料拆分。
- 营养字段解析。
- 我的关注项。
- 食品标签解读报告。
- 历史记录。
- 数据来源说明。

后续再做：

- 包装正面卖点核对。
- 两款商品对比。
- 扫码。
- 云同步。
- AI 总结。
- 小程序 / App。
- 订阅 / 支付。

## 11. 与数据可信的关系

消费者报告默认服务普通人决策，数据可信和法规证据仍必须存在，但应放在“数据来源和查看依据”中逐层展开。

- 普通用户默认看到：关注项、配料数量、添加剂数量、糖/钠/脂肪/蛋白质等标签信息、需要留意的原因。
- 需要时展开看到：数据来源、可信等级、GB2760 / JECFA / 人工复核 / OCR 来源说明。
- `pending_review`、`unknown_from_ocr`、AI 解释、OCR 原文都不能包装成权威结论。
