# AI UI Review Prompt

> 用途：真实 AI Provider 接通后，把成分镜微信小程序页面交给 AI 做逐页体验审查。AI 只做产品、UI 和文案建议，不改写事实、不生成监管或医疗结论。最终改动仍由本地代码、视觉烟测和产品输出审查验证。

## 审查目标

让普通用户拍食品标签后，尽快得到可理解、有价值、可行动的结果：

1. 一句话结论是否短、直接、非医疗化。
2. 为什么：热量、糖、钠、脂肪、添加剂、过敏原。
3. 建议关注人群：儿童、控糖、减脂、低钠、过敏 / 忌口。
4. 份量提示：只提示看每份 / 每包数字，不给吃法建议。
5. 添加剂解释：别吓人，也别洗白。
6. 识别详情：配料表、营养表、商品身份、包装声明、未确认线索。

## 全局红线

- 不让用户必须自己对照专业术语才能理解结果。
- 不堆砌专业数据，不把报告写成检测报告。
- 不把成分搜索变成首页主路径，只保留小入口。
- 不恢复历史 / 收藏入口；本期微信小程序不注册这些页面。
- 不恢复“建议吃法”大段说明。
- 不让 AI 覆盖本地规则的 `decision`、`riskLevel`、`nutritionJudgement`。
- 不输出医疗诊断、法规断言或绝对化风险词。

## 当前页面清单

### 首页

文件：`user-uniapp/src/pages/index/index.vue`

当前功能：
- 首屏只有一个主操作：拍包装。
- 点击后直接拉起相机；相机不可用才进入拍照页。
- 成分搜索只作为小入口“查单个成分”。

审查问题：
- 首页是否以拍包装为主，不像落地页或成分百科。
- 搜索入口是否足够弱，不抢主路径。
- 底部只保留首页 / 我的，是否清楚。

### 拍照 / 识别页

文件：`user-uniapp/src/pages/capture/index.vue`

当前功能：
- 支持拍照识别、上传识别、扫条码 / 二维码、手动输入。
- 清晰 OCR / AI 公开标签线索生成参考报告。
- 低置信、失败或信息不足生成信息不足报告。
- 用户主动手动输入或从报告补充时才进入补充信息页。

审查问题：
- 拍照 / 上传 / 扫码 / 手动输入按钮是否清楚。
- 图片选择失败提示是否只在选择失败时出现。
- 补充信息页是否聚焦配料表和营养数字，是否避免“需要确认”大段文案。

### 报告页

文件：`user-uniapp/src/pages/report/index.vue`

当前功能：
- 顶部展示识别信息卡。
- 展示一句话结论、用户关注短标签、为什么 / 关注人群 / 份量提示。
- 展示关键原因、建议关注人群、单列营养重点图、添加剂解释、识别详情。
- 信息不足报告只给补拍配料 / 营养表和手动粘贴文字两个动作。
- 本期不展示历史、收藏或建议吃法入口。

审查问题：
- 首屏是否能立刻回答“这次重点看什么”。
- 关键原因是否足够具体，不是泛泛提醒。
- 营养数字是否全部展示，并按用户关注或高含量突出。
- 信息不足是否不硬给消费结论，并明确缺少什么。
- AI 公开标签线索是否标注清楚，不伪装成包装 OCR、法规或医疗结论。

### 我的页

文件：`user-uniapp/src/pages/attention/index.vue`

当前功能：
- 关注目标支持多选：控糖、减脂、低钠 / 少盐、儿童零食。
- 过敏 / 忌口支持多选。

审查问题：
- 多选状态是否一眼可懂。
- 报告影响提示是否短、具体。

### 成分搜索页

文件：`user-uniapp/src/pages/search/index.vue`

当前功能：
- 数据库结果和 AI / 公开网页补充入口并列。
- 单个成分只作名称参考，不能替代整包包装报告。

审查问题：
- 数据库结果与 AI 补充是否区分清楚。
- 是否仍像成分百科或专业法规查询。
- 文字是否过长，是否影响快速参考。
