# AI UI Review Prompt

> 用途：真实 AI Provider 接通后，把成分镜微信小程序的每个页面功能交给 AI 做逐页体验审查。AI 只做产品和文案建议，不改写事实、不生成监管或医疗结论。最终改动仍由本地代码、视觉烟测和产品输出审查验证。

## 审查目标

让普通用户拍食品标签后，尽快得到可理解、有价值、可行动的结果：

1. 能不能买 / 能不能吃。
2. 为什么：热量、糖、钠、脂肪、添加剂、过敏原。
3. 适合谁 / 不适合谁。
4. 添加剂解释：别吓人，也别洗白。
5. 建议吃法：一次吃多少，怎么搭配。
6. 识别详情：配料表、营养表、生产日期。

## 全局红线

- 不让用户自己“对照包装原文”才能理解结果。
- 不堆砌专业数据，不把报告写成检测报告。
- 不出现“结果摘要 / 分享摘要”两层重复结构。
- 不出现“添加剂大白话”作为页面标题，统一用“添加剂作用”。
- 不把反馈做成大卡片，反馈只能是小入口。
- 不恢复“设置与说明”页面。
- 不把成分搜索变成首页主路径，只保留小入口。
- 不让 AI 覆盖本地规则的 `decision`、`riskLevel`、`nutritionJudgement`。
- 不输出医疗诊断、法规断言或绝对化风险词。

## 当前页面清单

### 首页

文件：`user-uniapp/src/pages/index/index.vue`

当前功能：
- 首屏只有一个主操作：拍配料表。
- 点击后进入 `pages/capture/index?auto=camera`，直接拉起相机。
- 成分搜索只作为小入口“查单个成分”。
- 页面内部不再放品牌标题，避免和微信导航标题重复。

审查问题：
- 主按钮文案是否比“拍配料表”更适合普通人？
- 首页是否足够简单，是否还存在不必要说明？
- 搜索入口是否足够弱，不抢主路径？

### 拍照 / 识别页

文件：`user-uniapp/src/pages/capture/index.vue`

当前功能：
- 支持相机、相册、手动输入。
- `auto=camera` 会直接打开相机。
- OCR 成功且有可用标签文字时自动生成结果。
- OCR 失败、低置信或手动场景进入结构化确认页。
- 确认页按配料表、营养数字、致敏原提示、包装声明拆分。
- 轻量识别质量提示只在有问题时展示。

审查问题：
- 自动识别失败时，用户是否知道下一步怎么做？
- 确认页是否仍然显得太重？
- 提示是否短、具体、可执行？
- 是否减少了用户自己整理 OCR 文本的负担？

### 结果页

文件：`user-uniapp/src/pages/report/index.vue`

当前功能：
- 顶部展示“一句话结论”。
- 展示关键原因、配料表里容易漏看的点、营养数字、适合谁 / 不适合谁、添加剂作用、建议吃法、识别详情。
- 识别详情包含商品、类型、生产日期、配料表、营养表、未确认线索、识别文字。
- 反馈在“识别文字”右侧作为小入口展开。
- 分享不再作为单独重复摘要卡。

审查问题：
- 首屏是否能立刻回答“这包能不能买 / 能不能吃”？
- 关键原因是否足够具体，不是泛泛提醒？
- 营养数字是否用图表展示得更直观？
- 添加剂解释是否中性、易懂、有价值？
- 建议吃法是否具体到份量和搭配？
- 是否还有重复模块或低价值文案？

### 历史记录

文件：`user-uniapp/src/pages/history/index.vue`

当前功能：
- 展示本机报告记录。
- 支持全部、收藏、关注项筛选。
- 卡片点击打开报告。
- 收藏、删除作为小操作。

审查问题：
- 列表密度是否合适？
- 卡片是否突出用户回看时最关心的信息？
- 操作按钮是否干扰阅读？

### 我的

文件：`user-uniapp/src/pages/attention/index.vue`

当前功能：
- 只承载关注设置：关注目标、儿童模式、过敏 / 忌口。
- 不再并列历史记录和成分查询入口。

审查问题：
- “我的”是否更像个人偏好设置，而不是功能堆叠页？
- 关注目标、儿童模式、过敏 / 忌口是否足够清楚？
- 设置保存反馈是否轻量？

### 成分搜索

文件：`user-uniapp/src/pages/search/index.vue`

当前功能：
- 作为辅助入口查询单个成分。
- 搜索结果强调为什么看、标签写法、数据来自、结果用途。
- 不给购买或医疗结论。

审查问题：
- 搜索结果是否足够普通人可理解？
- 是否避免了专业字段堆砌？
- 是否明确它是辅助功能，不是主路径？

## 输出格式

AI 审查时按下面格式返回，不要输出泛泛建议：

```text
页面：
当前问题：
为什么影响用户：
建议改法：
优先级：P0 / P1 / P2
是否需要改代码：是 / 否
验收方式：
```

## 执行顺序

1. 先审结果页，因为它决定产品是否有用。
2. 再审拍照 / 识别页，因为它决定能否顺利得到结果。
3. 再审首页，因为它决定用户是否知道从哪里开始。
4. 最后审历史、我的、成分搜索。

## 本地验收命令

每次根据 AI 审查改完 UI 后至少运行：

```bash
npm.cmd --prefix user-uniapp run typecheck
npm.cmd --prefix user-uniapp run lint
npm.cmd --prefix user-uniapp run audit:product-output
npm.cmd --prefix user-uniapp run regression:ocr-report
npm.cmd --prefix user-uniapp run build:mp-weixin
npm.cmd --prefix user-uniapp run build:h5
npm.cmd --prefix user-uniapp run visual:smoke
git diff --check
```

需要 ChatGPT/OpenAI 复审时，先确保截图已由 `visual:smoke` 生成，再运行：

```bash
npm.cmd --prefix user-uniapp run review:chatgpt
```

该命令只从运行环境读取 `OPENAI_API_KEY`，输出写入 `/tmp/complens-chatgpt-ui-review.json`。未配置 Key 时必须明确失败为 `missing_openai_api_key`，不得把本地推断伪装成 ChatGPT 评审。

真实 OCR 或真实 AI 相关改动还要补后端对应测试和服务联调记录。
