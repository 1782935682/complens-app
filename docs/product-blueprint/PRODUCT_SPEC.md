## 1. 产品定位

CompLens / 成分镜 **不是专业法规查询系统，也不是普通添加剂数据库工具**，而是一个**面向普通消费者的食品标签拍照解读与消费决策助手**。

用户的起点是手里那包食品和它的包装标签，而不是一个搜索框。产品价值在于：拍一下食品包装，帮普通消费者看懂**配料、营养、包装卖点和需要关注的地方**，形成一份可保存、可回看的食品标签解读。

核心用户问题：

- 这个食品适不适合我当前的关注目标？
- 我买之前需要看哪些点？
- 这款和另一款比，哪个更符合我的关注目标？

### 核心主路径（不可偏离）

```
拍照/上传食品标签
  → 自动识别标签类型：配料表 / 营养成分表 / 包装正面 / 未知
  → OCR 识别文字
  → 用户确认和修正识别文本
  → 配料拆分 / 营养字段解析 / 包装卖点识别
  → 匹配食品成分、食品添加剂、营养字段和用户关注项
  → 生成食品标签解读报告
  → 保存历史记录和产品档案
```

### 后续单一拍照入口自动识别模型（P1/P2）

当前 MVP 仍优先打磨拍食品标签 OCR 主路径；后续支持商品条码和数字标签二维码时，用户端仍保持一个拍照入口，原有首页 / 拍照页 UI 不需要改成三个入口。图片提交后由后端自动识别主要对象：

```text
拍食品标签 / 包装
  → 后端自动识别：商品条码 / 数字标签二维码 / 配料表 OCR / 营养表 OCR / 未知
  → 商品条码：根据 GTIN 查询商品及配料数据
  → 数字标签二维码：解析数字标签页面中的配料和营养信息
  → 配料表 OCR：直接识别包装上的配料表文字
  → 成分归一化
  → 官方成分知识库匹配
  → 成分分析
```

商品条码、数字标签二维码和配料表 OCR 是后端识别后的取数分支，不是三个外露用户入口。当前分支无法获得完整配料时，只提示用户可以重新拍摄包装其他区域或手动补充，不自动跳转，也不把三种识别方式设计成串行流程。

### 关键边界

- **成分搜索是辅助功能，不是主路径。** 搜索用于补充查询单个配料，不能替代“拍食品标签 → 食品标签解读”的完整流程，也不应在信息架构上抢占主入口。
- **专业法规和数据库信息默认隐藏。** 普通人报告默认展示关注项、配料/营养重点和购买前建议关注；法规证据、数据来源和原文依据放在“数据来源和查看依据”中展开。
- **当前阶段只做食品标签里的配料、食品添加剂、营养字段和包装卖点解读**，不混入化妆品、护肤品、药品。任何跨品类需求都属于未来探讨范围，不进入当前规范。
- 产品提供的是**信息参考**，不是健康结论或医疗建议（详见第 6 节非目标与全局文案约束）。
- **正式用户端技术路线**：`user-uniapp`，uni-app + Vue3，用于 H5/PWA、微信小程序、Android、iOS。当前纯 JS + Vite 前端仅作为历史原型和迁移来源，不继续承载复杂新功能。
- **后台管理端技术路线**：`admin-web`，Vue3 + TDesign Web，定位为产品运营后台 + 数据治理后台 + 系统配置后台。
- **后端 API 是所有端唯一数据入口**：OCR / AI / 数据库 / GB2760 导入复核 / 支付订阅均不得由用户端直连。

### 全局文案约束（贯穿全产品）

| 类型 | 内容 |
|---|---|
| 禁止文案 | 可以买、不能买、健康、不健康、安全、有害、致癌、治疗、诊断、一定过敏 |
| 推荐文案 | 建议关注、更适合重点查看、部分人群可能需要留意、信息不足，建议结合包装原文确认、仅供标签信息参考，不构成医疗或营养诊断 |

所有面向用户的文案、报告结论、提示语都必须遵守上表，详细判定口径见 [`DATA_TRUST_SPEC.md`](./DATA_TRUST_SPEC.md) 与 [`PRIVACY_AND_COMPLIANCE_SPEC.md`](./PRIVACY_AND_COMPLIANCE_SPEC.md)。

## 2. 目标用户

产品面向普通消费者，而非营养师、食品法规专员或专业数据库使用者。所有用户共性是：**看得到食品标签，但不知道买之前该重点看什么。**

| 用户群体 | 核心痛点 | 主要诉求 |
|---|---|---|
| 普通消费者 | 配料表、营养表和包装卖点分散，不知道哪些值得在意 | 拍一下就能看到购买前建议关注 |
| 家长 | 给孩子买零食时，担心甜味剂、色素、防腐剂、咖啡因 | 快速看到家长可能会关注的成分 |
| 过敏 / 忌口人群 | 配料里隐藏的过敏原（坚果、麸质、奶、蛋等）不易发现 | 自动标出可能的过敏原 / 忌口成分，给出提醒而非绝对判断 |
| 减脂 / 控糖 / 低钠人群 | 糖、钠、脂肪、蛋白质等信息散在配料表和营养表中 | 按关注目标聚合提示，辅助自我管理 |
| 少添加 / 配料简单人群 | 想知道配料数量、添加剂数量和复杂度 | 给出数量、分组和“建议查看”提示，不做健康判断 |
| 想对比两款食品的人 | 不知道哪款更符合自己的关注目标 | 对比糖、钠、脂肪、蛋白质、添加剂和关注项命中 |

> 注：以上为产品定位层的用户画像。各群体的功能优先级与界面表达详见 [`UI_ROADMAP.md`](./UI_ROADMAP.md) 与 [`FRONTEND_SPEC.md`](./FRONTEND_SPEC.md)。

## 3. 核心场景

以下场景均围绕“食品标签解读 + 消费决策辅助”展开，详细定义见 [`CONSUMER_DECISION_SPEC.md`](./CONSUMER_DECISION_SPEC.md)。

1. **超市 / 便利店购买前快速判断**：手里拿着食品，拍包装、配料表或营养成分表，快速知道糖、钠、甜味剂等是否值得关注。
2. **控糖人群看食品**：识别白砂糖、糖浆、浓缩果汁、甜味剂、碳水化合物和糖，不输出糖尿病人可以吃 / 不能吃。
3. **低钠人群看食品**：识别钠、食用盐、谷氨酸钠、碳酸氢钠、磷酸盐、酱油粉和调味料。
4. **给孩子买零食**：提示家长可能关注甜味剂、色素、防腐剂、咖啡因，不输出儿童不能吃 / 不适合儿童。
5. **过敏 / 忌口人群**：提示疑似乳、大豆、坚果、麸质、蛋、海鲜和用户自定义忌口项。
6. **少添加 / 配料简单人群**：展示配料数量、食品添加剂数量和配料复杂度。
7. **两款商品对比（后续）**：对比糖、钠、脂肪、蛋白质、添加剂数量和用户关注项命中。

每个场景的产出都应回到同一形态：一份**普通人优先、专业依据可展开**的食品标签解读，而非一句简单的“好 / 坏”结论。

## 4. MVP 范围（结合当前真实进度）

MVP 重新定义为：

```text
拍照/上传食品标签 → 标签类型识别 → OCR/手动输入 → 文本确认
→ 配料拆分 / 营养字段解析 → 我的关注项 → 食品标签解读报告
→ 保存历史记录和产品档案
```

下表按新主路径顺序列出必含能力，并标注当前真实状态。

状态口径：**已完成** = 主路径闭环可用；**进行中** = 已具备但仍在打磨或存在已知后置项；**待完善** = 计划内但尚未达到生产要求。

MVP 必须包含：
- 配料表识别。
- 营养成分表识别。
- 文本确认。
- 配料拆分。
- 营养字段解析。
- 我的关注项。
- 食品标签解读报告。
- 历史记录。
- 数据来源说明。

| # | MVP 必含能力 | 状态 | 说明 |
|---|---|---|---|
| 1 | 拍照 / 上传食品标签 | 已完成 | 当前入口已支持图片采集；文案和流程需从“配料表”扩展为“食品标签”。 |
| 2 | 标签类型识别 | 待完善 | 需识别 `ingredient_list` / `nutrition_facts` / `front_claims` / `unknown_label`；无法判断时允许用户选择。 |
| 3 | 配料表识别 | 进行中 | OCR manual/mock/本机 RapidOCR 闭环已完成；生产 Aliyun OCR 后置。 |
| 4 | 营养成分表识别 | 待完善 | 新增 MVP 能力，需解析能量、蛋白质、脂肪、碳水、糖、钠等字段。 |
| 5 | 文本确认 | 已完成 / 待扩展 | 当前配料文本确认已具备；需支持营养表和包装正面文案确认。 |
| 6 | 配料拆分 | 已完成 | 对确认后的配料表文本进行解析与拆分。 |
| 7 | 营养字段解析 | 待完善 | 新增 MVP 能力，结构化 `energy/protein/fat/carbohydrate/sugar/sodium` 等字段。 |
| 8 | 我的关注项 | 进行中 / 待提升 | 当前偏好能力已有基础；需升级为主路径核心，本地保存即可，不要求登录。 |
| 9 | 食品标签解读报告 | 待完善 | 报告页需改为普通消费者结构：一句话结论、关注项、购买前建议关注、配料/营养/卖点/来源。 |
| 10 | 历史记录 | 已完成 | 保存产品档案、收藏、历史；需适配食品标签解读报告。 |
| 11 | 数据来源说明 | 已完成 | 数据来源和可信等级保留，但专业信息默认折叠到“查看依据”。 |

### 方向修订补充（2026-06，覆盖优先级）

依据 [`PRODUCT_DIRECTION_2026H2.md`](./PRODUCT_DIRECTION_2026H2.md)：

- **新增核心 P0**：**添加剂大白话翻译**——把"脱氢乙酸钠 / 卡拉胶 / 三氯蔗糖"翻成"防腐剂 / 增稠剂 / 代糖 + 一句人话作用 + 是否常见"。用一份精选词表（约 150–300 条）即可覆盖约 90% 场景，**不依赖 AI、不依赖全量 GB2760**；已有 GB2760 / 成分库成果转为该词表来源。
- **第一版收敛为微信小程序单端**；H5 / App 扩端在小程序验证成功后再做。
- **降级 / 暂停**：GB2760 全量导入与 7 级可信治理、admin-web 后台、专业法规查询入口、旧 `src/` 原型维护——详见方向修订文档第 5 节。
- **OCR 保留自建 RapidOCR**（本次决策）。

### 真实数据状态枚举（数据来源展示的取值）

报告与匹配结果必须使用以下受控枚举标注每条成分的可信状态，不得自创口径：

| 枚举值 | 含义 |
|---|---|
| `verified_regulation` | 官方标准已验证 |
| `verified_jecfa` | 安全评价已匹配（非中国法规范围） |
| `pending_review` | 待复核来源数据 |
| `mapped_candidate` | 疑似匹配，待确认 |
| `common_ingredient` | 普通配料 |
| `unverified` | 未验证 |
| `unknown_from_ocr` | 暂未收录 |

枚举的展示规则、颜色 / 标签与降级表达详见 [`DATA_TRUST_SPEC.md`](./DATA_TRUST_SPEC.md)。

## 6. 非目标（明确不做）

为守住产品边界与合规底线，以下事项明确**不做**：

- **不做医疗诊断。** 产品不判断疾病、不提供诊疗意见。
- **不做疾病治疗建议。** 不出现任何“治疗 / 缓解 / 改善病症”类表述。
- **不做绝对健康 / 绝对有害判断。** 不输出“健康 / 不健康 / 安全 / 有害 / 致癌 / 可以买 / 不能买 / 一定过敏”等结论（见第 1 节禁止文案）。
- **不把 AI 生成内容作为权威数据源。** AI 仅用于辅助解释 / 归纳，权威性以官方标准来源与可信等级为准。
- **不把 OCR 结果直接作为最终结论。** OCR 文本必须经用户确认修正后才进入后续分析，识别结果不等于配料事实。
- **不跨品类。** 当前不涉及化妆品、护肤品、药品。

所有输出统一以“仅供标签信息参考，不构成医疗或营养诊断”为基调，鼓励“建议关注 / 更适合重点查看 / 部分人群可能需要留意 / 信息不足，建议结合包装原文确认”这类非绝对化表达。

## 7. 待确认事项

以下为需要产品负责人 / 用户拍板的边界问题，确认前在相关文档中标注“待确认”，不擅自实现：

1. **原图留存策略**：是否长期保存用户上传的配料表原图？保存时长、是否可一键删除、是否仅保存识别文本而不保存原图——待确认。
2. **付费点设计**：订阅 / 一次性解锁 / 免费额度的边界（如每月识别次数、AI 高级分析是否付费、历史档案容量上限）——待确认。
3. **登录与匿名使用边界**：匿名用户可使用到哪一步、哪些功能必须登录（如云同步、跨设备历史）——待确认。
4. **生产 OCR 供应商最终选型**：Aliyun 是否为最终生产方案，是否需要多供应商冗余，本机 RapidOCR 是否保留为离线降级——待确认。
5. **真实 AI 分析的范围与定位**：AI 在报告中扮演“解释助手”还是“关注点归纳”，输出是否需明确标注“AI 生成、非权威来源”——待确认（基调已定：不作为权威数据源）。
6. **过敏原 / 忌口清单来源**：使用预置标准过敏原清单，还是允许用户自定义忌口词条，二者如何合并提示——待确认。
