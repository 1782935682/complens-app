# DESIGN_GUIDE.md

# 配料雷达微信小程序 UI / UX 设计规范

本文档定义「配料雷达」当前消费者端小程序的 UI、文案、页面和消费建议体验规范。适用范围为 `user-uniapp/` 微信小程序用户端页面与组件。

---

## 1. 产品定位

「配料雷达」是一个帮助普通消费者在买食品、饮料、零食、酸奶和调味品前快速看懂配料表的消费决策助手。

一句话表达：

> 拍一下，看懂配料表。

核心场景：

- 给孩子买零食时，快速看看是否适合高频食用。
- 减脂、控糖用户重点关注糖、甜味剂、脂肪、钠等信息。
- 对牛奶、花生、鸡蛋、大豆等成分敏感的人，快速检查包装文字。
- 添加剂识别是核心能力，配料解释是理解过程，营养分析是辅助判断，个性化消费建议是最终输出。

产品不是专业检测工具、医疗 App、营养诊断工具、食品安全评级系统或后台管理工具。MVP 暂不混入日化规则。

---

## 2. 设计关键词

整体体验必须符合：

- 简单：首屏只保留最重要操作。
- 清爽：少边框、少装饰、少密集信息。
- 亲和：用生活化短句，不制造焦虑。
- 轻量：像购物辅助工具，不像检测报告。
- 易懂：普通用户不需要理解专业术语也能继续。
- 决策优先：先给消费建议，再解释原因。

避免方向：

- 专业检测报告
- 医疗健康 App
- 风险评估系统
- 企业后台管理工具
- 复杂数据看板
- 扫描仪式工具界面

## 3. 色彩规范

### 3.1 主色

- 主色：清新蓝绿色 `#129780`
- 强调主色：深蓝绿色 `#087a68`
- 浅底主色：薄荷浅绿 `#eefaf5`
- 辅助高亮：明亮绿 `#2fc89f`

使用规则：

- 主按钮、当前步骤、可点击重点入口使用主色。
- 大面积背景使用浅底主色或米白，不使用高饱和纯色。
- 图表和标签可使用柔和橙、青绿、浅灰分层。

### 3.2 背景与卡片

- 页面背景：米白灰 `#f7f6f1`
- 卡片背景：白色 `#ffffff`
- 轻卡片背景：浅绿白 `#f7fcf8`
- 温和提示背景：奶油色 `#fff8ec`

页面应该有空气感，避免大片纯白或大片深色。

### 3.3 文字

- 主标题：深墨绿 `#17211d`
- 正文：深灰绿 `#37423e`
- 辅助文字：灰绿 `#6a756f`
- 弱说明：浅灰 `#8a938e`

### 3.4 提醒色

- 轻提醒：柔和橙 `#d88a24`
- 轻提示底：浅橙 `#fff4df`
- 信息提示：青蓝绿 `#0e8a96`
- 删除/危险操作：柔和珊瑚 `#d96b5f`

禁止大面积使用：

- 纯红
- 大面积黑色
- 医疗蓝
- 强警告色
- 高饱和荧光色

## 4. 字体与排版

### 4.1 字体

默认使用系统字体：

`-apple-system, BlinkMacSystemFont, PingFang SC, HarmonyOS Sans SC, MiSans, Microsoft YaHei, sans-serif`

### 4.2 字号

- 页面标题：24px，粗体，短句。
- 卡片标题：18px，粗体。
- 正文：16px，行高 1.6。
- 辅助说明：14px，行高 1.6。
- 标签文字：12px，粗体。

### 4.3 间距

- 页面左右留白：16px。
- 卡片内边距：16px。
- 卡片之间间距：16px。
- 模块之间间距：20-24px。
- 按钮最小高度：46px。

小程序页面要避免信息太满。每个卡片只表达一个主题。

## 5. 组件规范

### 5.1 卡片

用途：

- 首页功能入口
- 消费建议
- 我的目标/忌口
- 扫描记录
- 隐私说明
- 免责声明

样式：

- 圆角统一 16px。
- 使用轻阴影或 1px 柔和边框。
- 背景干净，优先白色。
- 不使用复杂边框和强烈投影。
- 卡片内不堆多层卡片。

### 5.2 按钮

主按钮：

- 用于「拍配料表」「生成解读」等关键动作。
- 颜色明显，使用主色。
- 文案短，推荐 4-8 个字。
- 圆角略大，适合小程序触摸。

次按钮：

- 用于「从相册选择」「复制原文」「返回报告」等辅助动作。
- 样式比主按钮弱。

危险按钮：

- 只用于清空历史、清空关注项等操作。
- 使用柔和红，不使用强红。
- 必须有二次确认。

### 5.3 标签

用于显示：

- 常见过敏原
- 甜味剂
- 防腐剂
- 色素
- 咖啡因
- 用户关注词

标签文案：

- 重点关注
- 可以留意
- 普通提示
- 需核对
- 添加剂

禁止使用：

- 高风险
- 危险
- 有害
- 不能吃
- 食品安全评级

### 5.4 空状态 / 加载态 / 错误态

空状态：

- 说明发生了什么。
- 给一个清晰下一步。
- 不使用恐吓或专业术语。

加载态：

- 使用轻骨架或短文案。
- 避免长时间占满屏幕。

错误态：

- 解释为「暂时不可用」或「请重试」。
- 保留可继续路径，如重新上传、手动输入。

## 6. 页面规范

### 6.1 首页

目标：用户 3 秒内知道这是“拍配料表 / 营养成分表，看懂食品标签”的工具。

结构：

1. 顶部品牌区
   - 标题：拍配料表，看懂成分
   - 副标题：支持配料表和营养成分表，识别添加剂、糖、钠、脂肪和过敏原。
2. 主操作区
   - 只保留一个最强按钮：拍食品标签
   - 手动输入作为弱入口
3. 我的关注摘要
   - 展示当前关注，例如控糖、少盐、儿童、过敏/忌口
4. 最近扫描
   - 展示最近 3 条
   - 没有记录时显示空状态
5. 今日提示
   - 糖、钠、油脂、配料顺序等生活化提示
6. 隐私提示
   - 文案：识别内容仅用于本次分析，历史记录可清除

首页不要像功能菜单，不展示数据来源、成分详情、专业说明页，不把商品对比、商品正面识别、条码识别作为核心入口。

### 6.2 拍食品标签页

目标：让用户用拍照、相册或手动输入完成主流程，不被 OCR 失败卡住。

页面重点：

- 大面积上传区。
- 主按钮「开始 OCR 识别」。
- 次按钮「从相册选择」。
- 手动输入入口与拍照同级可见。
- 拍摄对象只写配料表和营养成分表。
- 拍照区展示透明矩形引导框，提示用户把配料表 / 营养成分表放在框内，减少广告语和生产信息进入 OCR。
- 不在 P0 承诺商品正面、条码或商品库命中。
- OCR 成功后进入轻确认，不直接生成结果。
- 识别失败时直接提示改为手动输入。

推荐文案：

- 尽量拍完整配料表或营养成分表。
- 文字清楚、少反光，识别会更准。
- 识别后还可以手动修改文字。

### 6.3 文本确认区

目标：让用户知道文字可修改，不担心识别错误。

页面重点：

- 不展示复杂置信度。
- 不让用户选择专业标签类型。
- 先展示识别摘要：添加剂、糖、钠、脂肪、蛋白质、过敏原是否命中。
- 轻确认只展示清洗后的有效标签文本：配料表、营养成分、致敏原提示；不要把整段 OCR 原文直接放到规则分析前。
- 文本框只作为可选修正，不要求全文逐字校对。
- 如果没有识别到清晰配料表，提示用户重新拍配料表区域或手动输入。
- 主按钮「确认无误，生成解读」。
- 次按钮「识别不准，手动修改」。

推荐文案：

- 识别文字可能有错，可以先改一下。
- 请尽量保持和包装上一致。

### 6.4 消费建议页

目标：用户 5 秒内看懂这个商品是否适合自己的目标。

结构：

1. 顶部结论
   - 适合当前目标
   - 可以偶尔选
   - 需要留意
   - 不太适合当前目标
2. 一句话建议
3. 你需要留意的点
4. 添加剂识别
5. 更多信息折叠区
   - 营养快照
   - 原始识别文本
   - 数据说明
   - 免责声明
6. 操作按钮
   - 复制结果
   - 再扫一个
   - 先放一边

示例文案：

- 可以偶尔选：可以结合你的目标控制频率和份量。
- 为什么提醒：配料中出现糖或糖浆类成分，控糖或减脂人群可以留意。
- 过敏原提醒：含有你关注的过敏原，建议先不要食用，并核对包装过敏原提示。

### 6.5 我的关注目标页

标题：我的关注

说明：只选一个主目标，再按需开启儿童模式和过敏/忌口。结果页会按这些目标调整提醒重点。

模块：

1. 主目标（单选）
   - 日常
   - 控糖
   - 减脂
   - 低钠/少盐
2. 儿童模式（独立开关）
   - 更重点提醒高糖、咖啡因、酒精、人工色素、甜味剂、高钠。
3. 过敏/忌口（多选）
   - 花生
   - 坚果
   - 牛奶
   - 大豆
   - 麸质
   - 鸡蛋
   - 鱼类
   - 虾蟹类

P0 暂时隐藏健身、敏感体质和自定义关注词。

### 6.6 扫描记录页

标题：扫描记录

空状态：

还没有扫描记录。拍一次配料表或营养成分表，结果会保存在这里。

列表展示：

- 扫描时间
- 本次结论
- 简短摘要
- 支持重新打开结果和删除记录

### 6.7 设置页

设置页只承载隐私、免责声明、清空历史和关于产品。不要向普通用户展示后端地址配置、商品库配置、规则库版本或开发调试项；如需调试入口，仅开发环境隐藏使用。

### 6.8 数据说明

数据说明不作为主流程页面，统一放在消费建议页底部。

内容要求：

- 默认折叠展示。
- 说明本次分析依据：用户拍摄的配料表、用户拍摄的营养成分表、手动输入内容或历史记录。
- 说明本次分析基于包装文字、营养字段、成分匹配和本地关注目标。
- 来源与规则版本独立展示，不混在结论卡片里。
- 未确认项只提供搜索线索和核对提示。
- 明确提示本工具仅作消费参考，不替代医生、营养师或监管机构意见。

### 6.9 成分解释

成分解释不作为独立主流程页面，只在消费建议页使用半屏弹窗承载。

页面重点：

- 标题使用“成分解释”。
- 用普通语言说明，不做恐吓式结论。
- 对常见添加剂、过敏关注、控糖/低钠/儿童等场景影响做简短解释。
- 入口要弱化，不抢占消费建议页主视觉。

### 6.10 商品对比页

目标：当用户在两个商品之间犹豫时，给出更偏向哪款及原因。

商品对比属于 P1，不作为 MVP 首页核心入口。

页面重点：

- 支持输入/拍摄商品 A。
- 支持输入/拍摄商品 B。
- 顶部先给“更推荐 A / 更推荐 B / 两者差不多”。
- 展示主要原因。
- 下方再展示糖、钠、能量、脂肪、添加剂、关注项、未识别项等对比。
- 暂时基于本地规则，不依赖真实商品数据库。

## 7. 文案规范

文案风格：

- 短句。
- 口语化。
- 友好。
- 不恐吓。
- 不专业。
- 不下结论。

推荐文案：

- 发现你关注的配料
- 这些成分可以留意一下
- 暂未发现你设置的关注成分
- 图片优先在本机识别
- 仅供配料阅读参考
- 添加你在意的配料，扫描时自动提醒

禁止文案：

- 有毒
- 危险
- 不能吃
- 高风险
- 不健康
- 食品安全评级
- 医疗建议
- 营养诊断
- 检测报告
- 风险评估
- 安全
- 有害
- 致癌
- 治疗
- 诊断

## 8. 微信小程序适配规范

- 页面不做 PC 化布局。
- 点击区域不小于 44px。
- 主要操作放在屏幕上半部分或页面底部清晰位置。
- 页面底部留出 `safe-area-inset-bottom`。
- 弹窗只用于确认类操作，不做复杂表单。
- 图片上传区高度稳定，避免跳动。
- 长文本默认折行，不横向滚动。
- 不使用前端暴露 OCR Key、AI Key 或数据库连接。

---

## 9. 验收清单

每轮 UI 修改完成后检查：

- [ ] 首页 3 秒内能理解产品用途。
- [ ] 主操作「拍配料表」足够明显。
- [ ] 页面没有专业报告感和后台系统感。
- [ ] 文案没有禁止词。
- [ ] 空状态、加载态、错误态都有下一步。
- [ ] 卡片、按钮、标签使用统一样式。
- [ ] 数据来源不挤在结果页里。
- [ ] 小程序构建通过。
