# Project Plan

## 1. 项目最终目标

CompCheck（成分小查）的目标是完成一个可正式使用、可持续迭代的成分查询与分析 App。

最终产品范围包括：

- 成分搜索、详情、分类、别名、INCI/英文名查询。
- 产品成分表文本解析和图片 OCR 识别。
- 基于可靠数据源的成分风险提示、适用提醒、功效说明和免责声明。
- 收藏、历史、用户设置，必要时支持云端同步。
- 后端 API、数据库、第三方 OCR/AI 代理、密钥管理、日志和限流。
- 数据采集、审核、版本管理、来源追踪和持续维护流程。
- 移动端友好 UI、可访问性、错误状态、加载状态和真实设备验收。
- 自动化测试、CI、部署、监控、隐私政策和上线合规。

## 2. 当前阶段

当前阶段：第一阶段 / 早期静态前端原型

当前判断：项目刚开始。现在只能算“前端原型已起步”，不能算产品主体已经完成。

- 当前测试只覆盖少量本地服务函数，没有 UI 自动化、端到端测试、真机验收和部署验收。

## 3. 真实进度口径

之前写成 72% 不准确。原因是把“页面入口做出来”和“完整产品能力完成”混在了一起。

本计划之后统一按“完整可上线产品”计算进度，包含数据完善、后端、OCR、AI、测试、部署、合规和长期维护。

整体产品进度：4%

这个 4% 是保守估算：当前代码已经有静态前端原型和少量本地逻辑，但核心资产“数据”和核心能力“OCR/AI/后端/上线体系”基本未开始。

## 11. 维护规则

每次编码 Agent 修改项目后，必须同步检查并更新 `PROJECT_PLAN.md`。

如果本次修改影响了项目目标、阶段进度、已完成功能、未完成功能、下一步计划或验收状态，必须同步更新 `PROJECT_PLAN.md`。

如果本次修改完成了某个功能模块，必须更新对应模块状态和进度。

后续更新进度时必须遵守：

- 不允许只按页面完成度计算总进度。
- 总进度必须包含数据、后端、OCR、AI、测试、部署、合规。
- “入口已预留”不能写成“功能已完成”。
- “本地示例数据可用”不能写成“数据体系完成”。
- 每个完成度必须能对应到代码、测试、文档或验收记录。
